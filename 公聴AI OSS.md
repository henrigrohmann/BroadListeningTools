バックエンドとフロントエンドの役割分担は？ また、ロジックは両方にまたがってるのか、など相互依存の程度についてざっとまとめて。

---
バックエンドとフロントエンドの役割分担はかなり明確で、ロジックが両方にまたがる部分は最小限に抑えられています。ただし、レポート生成という性質上、データ構造の共有や処理フローの依存は存在するという構造です。

---
バックエンドの役割（FastAPI / Python）

🎯 中心となる責務

• レポート生成パイプラインの実行• LLM（OpenAI / ローカルLLM）を使った要約・分類・クラスタリング
• テキスト処理や分析ロジックはすべてバックエンド側
• データ管理• レポート、セクション、クラスタ、コメントなどの永続化
• DBアクセス（ORM）
• API提供• admin / public-viewer が利用する REST API
• 非同期処理（タスク実行）• レポート生成のような重い処理をバックグラウンドで実行

🔍 特徴

• アプリの「頭脳」部分はほぼバックエンドに集中
• フロントは「表示」と「操作指示」を行うだけで、分析ロジックは持たない

---
フロントエンドの役割（Next.js / TypeScript）

🎯 public-viewer（閲覧側）

• レポートの可視化（グラフ、クラスタ、要約）
• 静的サイトとしての配信も可能（GitHub Pages など）

🎯 admin（管理側）

• レポート作成 UI
• コメントのアップロード
• パイプライン設定（どの分析を行うか）
• バックエンドのタスク実行状況の表示

🔍 特徴

• ロジックは UI ロジックに限定• 入力フォームのバリデーション
• グラフ描画
• API から受け取ったデータの整形・表示
• 分析ロジックは一切持たない

バックエンドとフロントの相互依存の程度

🔗 依存は「データ構造」と「API契約」に集中

• API のエンドポイント仕様
• レポート生成のステータス管理（pending / running / done）

🔍 ロジックがまたがる部分は？

ほぼない。

しかし、分析・分類・要約などの本質的ロジックは完全にバックエンド側。

🧭 レポート生成パイプラインの全体像（バックエンド中心）

レポート生成は、次のような段階で進みます。

1. コメントデータの投入（admin → API）
2. 前処理（API）
3. LLM による分類・クラスタリング（API）
4. クラスタごとの要約生成（API）
5. レポート構造の保存（API → DB）
6. public-viewer が完成データを取得して可視化（viewer → API）

この流れのうち、2〜5 はすべてバックエンド側で完結します。

---
🧩 各ステップの役割分担

1. コメント投入（admin）

• CSV/テキストをアップロード
• API に「レポート生成を開始して」と指示
• フロントの役割は UI 操作のみ

2. 前処理（API）

• テキストのクリーニング
• 重複除去
• トークン数の調整
• 完全にバックエンドのロジック

3. LLM による分類・クラスタリング（API）

• OpenAI / ローカルLLM（Ollama）を呼び出す
• コメントをクラスタに分ける
• クラスタ名（テーマ）を生成
• 分析ロジックの中核で、フロントは一切関与しない

4. 要約生成（API）

• 各クラスタの要約を LLM で生成
• レポート全体の総括要約も生成
• ここもバックエンドのみ

5. レポート構造の保存（API → DB）

• sections
• clusters
• summaries
• metadata
• データモデルはバックエンドが定義し、フロントはそれを読むだけ

6. public-viewer による可視化（viewer）

• グラフ描画（クラスタ数、コメント数など）
• クラスタごとの要約表示
• コメント一覧のフィルタリング
• UI ロジックのみで、分析ロジックは持たない

---
🔗 相互依存の程度（どれくらい結びついている？）

依存が強い部分

• データ構造（API のレスポンス形式）• レポートの JSON 構造
• クラスタのフィールド（id, label, summary, comments など）

• ステータス管理• pending / running / done
• admin はこれをポーリングして進捗を表示

依存が弱い部分

• ロジック（分析・分類・要約）
→ 完全にバックエンドに閉じている
• UI 表示
→ 完全にフロントに閉じている

またがるロジックは？

• ほぼ存在しない• フロントは「整形」「フィルタ」「グラフ描画」程度
• 本質的な分析ロジックは API 側に集約

   |                         └--(結果保存)--> [DB]

DBには何を使っているの？

---
結論として、広聴AI（kouchou‑ai）のバックエンドは SQLite を使っています。
これはローカル開発・小規模運用を前提とした、軽量でセットアップ不要のデータベースです。

---
🗄️ 使われているデータベース：SQLite

リポジトリの構成を見ると、バックエンド（api）は FastAPI + SQLAlchemy を使っており、
compose.yaml や .env.example では外部DB（PostgreSQL など）の設定が存在しません。

さらに、プロジェクトのディレクトリには server/db.sqlite3 が含まれており、
実際に SQLite ファイルが生成される構成になっています。

SQLite を採用している理由は次の通りです。

• Docker で簡単に動かせる
• 小規模データ（数千〜数万コメント）なら十分高速
• 外部DBのセットアップが不要
• 静的出力（public-viewer）と相性が良い

---
📦 DB に保存される主なデータ

SQLite には、レポート生成に必要な次のようなデータが保存されます。

• レポート情報（metadata）
• コメント（raw comments）
• クラスタ（LLM による分類結果）
• クラスタ要約
• レポート全体の要約
• レポート生成タスクのステータス

つまり、レポート生成パイプラインのすべての中間データが DB に保存されます。

---
🔄 将来的な拡張性（PostgreSQL など）

現状は SQLite ですが、FastAPI + SQLAlchemy の構成なので、

• PostgreSQL
• MySQL
• MariaDB
• Cloud SQL
• Azure Database

などへの移行は比較的容易です。

実際、自治体導入など大規模運用を想定する場合は、
PostgreSQL への移行が自然な選択になります。

---
🧭 まとめ

• 現状の DB：SQLite
• 用途：レポート生成の全データを保存

レポート生成で保存される主要データ

🗂️ レポート（Report）

レポート全体のメタ情報が入ります。

• レポート名・説明
• 作成日時・更新日時
• ステータス（pending / running / done / failed）
• 使用した LLM モデル名
• コメント総数
• セクション構造（例：テーマ別、カテゴリ別）

レポートは「親」データで、以下のセクション・クラスタ・要約がぶら下がります。

---
セクション（Section）

レポートを構造化するための中間レイヤー。

• セクション名（例：治安、子育て、インフラ）
• セクション説明
• セクション内のクラスタ一覧

自治体のレポートでよくある「章立て」に相当します。

---
クラスタ（Cluster）

LLM によって分類された「似た意見のまとまり」。

• クラスタ名（テーマ名）
• クラスタ要約（LLM生成）
• コメント数
• クラスタの代表文（代表的な意見）
• クラスタのスコア（類似度など）

クラスタはレポートの中核で、public-viewer で最も可視化される部分です。

---
コメント（Comment）

ユーザーがアップロードした自由記述テキスト。

• 元のコメント本文
• 前処理後のテキスト
• どのクラスタに属するか
• コメントの長さ・メタ情報

クラスタリング後は、コメントはクラスタに紐づけられます。

---
要約（Summary）

レポート全体やクラスタごとの要約。

• クラスタ要約（各クラスタの短い説明）
• レポート全体の総括要約
• LLM が生成した補助的な説明文

---
タスク（Task / Pipeline status）

レポート生成の進捗管理。

• ステータス（pending / running / done / failed）
• 実行ログ
• エラー内容
• 実行時間

admin 画面がポーリングして進捗を表示するために使われます。

---
モデル設定・パイプライン設定

レポート生成時に使った設定も保存されます。

• 使用した LLM（OpenAI / ローカルLLM）
• プロンプトテンプレート
• クラスタ数の推定方法
• 前処理の設定

再現性のために重要なデータです。

---

SQLite に保存されているのは、次のような「レポート生成に必要なすべてのデータ」です。

• レポートのメタ情報

---
レポート生成パイプラインの「実際のテーブル構造（カラム一覧）」も見たい？

---
バックエンド処理におけるデータの流れ、そしてデータ加工および分析の流れをまとめて欲しい。

---
バックエンド処理のデータの流れは、**「コメントを受け取る → 前処理 → LLM分析 → 構造化 → 保存 → フロントへ提供」**という一本のパイプラインで動いています。
ここでは、どの段階でどんなデータが生成・加工され、どのように次の工程へ渡されるかを、処理フローに沿って体系的にまとめます。

---
1. コメント受け取り（入力段階）

役割

admin 画面からアップロードされた CSV や JSON を FastAPI が受け取る。

入力データの例

~~~
[
  { "id": 1, "text": "公園の遊具が古くて危ない" },
  { "id": 2, "text": "夜道が暗いので街灯を増やしてほしい" }
]
~~~

バックエンドで行われること

• バリデーション（空文字、重複、文字コードなど）
• DB に「生コメント」として保存
• レポート生成タスクを作成（pending 状態）

---
2. 前処理（クリーニング）

目的

LLM が扱いやすい形に整える。

主な処理

• 改行・記号の正規化
• 重複コメントの除去
• 文章のトークン数チェック
• 不要語の除去（ノイズ削減）

出力データ

~~~
[
  { "id": 1, "clean_text": "公園の遊具が古くて危ない" },
  { "id": 2, "clean_text": "夜道が暗いので街灯を増やしてほしい" }
]
~~~

---
3. LLM によるクラスタリング（分析の中核）

目的

似た意見をまとめ、テーマ（クラスタ）を作る。

処理内容

• コメントを LLM に渡し、類似度ベースでグループ化
• 各クラスタに「テーマ名」を生成
• コメント → クラスタ の紐付けを作成

出力データ例

~~~
clusters: [
  {
    "cluster_id": 1,
    "label": "公園・遊具の安全性",
    "comment_ids": [1]
  },
  {
    "cluster_id": 2,
    "label": "街灯・夜間の安全",

• レポート全体の総括要約も生成

⑤ レポート構造化（セクション化）

処理内容:

• セクション（章）を生成
• セクション内にクラスタを配置
• コメント → クラスタ → セクション → レポート の階層構造を構築

---

目的: レポートの再利用・再編集・静的出力を可能にする。

• reports（メタ情報）

OpenAI と Ollama の役割分担（要点）

• OpenAI• クラウドの高性能 LLM（GPT‑4 など）を使う
• 精度が高く、安定している
• コストがかかる
• API キーが必要
• GPU がなくても使える

• Ollama• ローカルで動く LLM（Llama‑3‑ELYZA‑JP‑8B など）
• 無料で使える
• GPU があれば高速
• モデルサイズに応じて精度は OpenAI より低いこともある
• オフライン環境でも動く

どちらも「クラスタリング」「要約」「セクション生成」などの LLM 推論を担当する。
ただし、どちらを使うかは設定で切り替えるだけで、処理フローは同じ。

分析ロジックではなく、UX 改善のためのデータ収集です。

🎨 どのライブラリが散布図などの描画に使われているか

1. Recharts（public‑viewer / admin）

広聴AIのフロントで最も使われているグラフ描画ライブラリは Recharts です。

• React 専用のチャートライブラリ
• 軽量で扱いやすい
• 散布図（ScatterChart）を含む多くのチャートに対応
• public‑viewer のクラスタ可視化や統計表示に利用される

対応チャート例：

• 散布図（ScatterChart）
• 折れ線グラフ（LineChart）
• 棒グラフ（BarChart）
• 円グラフ（PieChart）
• レーダーチャート（RadarChart）

2. D3.js（間接的に利用される可能性）

Recharts は内部的に D3.js を利用しています。

• スケール計算
• 軸のレンダリング
• レイアウト処理

ただし、プロジェクト側で直接 D3 を書くことはありません。

---
3. Next.js + React（描画の基盤）

チャートは Next.js の React コンポーネントとして描画されます。

• クライアントサイドで動的に描画
• public‑viewer の静的出力でも動作
• レポート JSON を受け取り、チャートに変換

---
🧠 バックエンド側で可視化ライブラリは使われていない理由

バックエンド（FastAPI）は以下の役割に限定されています。

• コメントの分析（LLM）
• クラスタリング
• 要約生成
• データ構造化
• SQLite への保存

グラフ描画はすべてフロント側に任せる設計になっているため、
Python 側では matplotlib / seaborn / plotly などは使われていません。

バックエンド API の全体構造

API は大きく次の 5 つの領域に分かれています。

1. レポート管理 API
2. コメント管理 API
3. クラスタ・要約データ API
4. レポート生成パイプライン API
5. システム・ユーティリティ API（ヘルスチェックなど）

FastAPI の router は /api/v1/... のようにバージョン付きで構成されます。

---
1. レポート管理 API

■ GET /reports

レポート一覧を取得。

レスポンス例

~~~
[
  {
    "id": 1,
    "title": "市民意見レポート",
    "status": "done",
    "created_at": "...",
    "updated_at": "..."
  }
]
~~~

---
■ POST /reports

新規レポートを作成（空のレポート枠だけ作る）。

リクエスト

~~~
{
  "title": "2024年度 市民意見レポート",
  "description": "アンケート自由記述の分析"
}
~~~

レスポンス

~~~
{ "id": 12, "status": "pending" }
~~~

---
■ GET /reports/{report_id}

レポートのメタ情報を取得。

---
■ DELETE /reports/{report_id}

レポートを削除（関連データも cascade）。

---
2. コメント管理 API

■ POST /reports/{report_id}/comments

コメントをアップロード（CSV/JSON）。

リクエスト例

~~~

    { "text": "公園の遊具が古い" },
    { "text": "夜道が暗いので街灯を増やしてほしい" }
  ]

~~~

レスポンス

~~~
{ "count": 2 }
~~~

---
■ GET /reports/{report_id}/comments

レポートに紐づくコメント一覧を取得。

---
3. クラスタ・要約データ API

■ GET /reports/{report_id}/clusters

クラスタ一覧を取得。

レスポンス例

~~~
[
  {
    "cluster_id": 1,
    "label": "公園の安全性",
    "comment_count": 12
  }
]
~~~

---
■ GET /reports/{report_id}/clusters/{cluster_id}

クラスタ詳細（コメント一覧・要約）を取得。

---
■ GET /reports/{report_id}/summary

レポート全体の総括要約を取得。

---
4. レポート生成パイプライン API（最重要）

■ POST /reports/{report_id}/generate

• クラスタリング（OpenAI or Ollama）
• クラスタ要約生成

レスポンス

API の特徴（設計思想）

• フロントは DB に触らず、すべて API 経由
• 分析ロジックはバックエンドに完全集約
• OpenAI / Ollama の切り替えは内部で抽象化
• レポート生成は非同期タスクとして実行
• public-viewer は完成した JSON を取得して描画するだけ

---
まとめ

広聴AIのバックエンド API は、次の 3 つの柱で構成されています。

1. レポート・コメント・クラスタの CRUD API
2. レポート生成パイプライン API（LLM を使う中核）
3. タスク管理・ヘルスチェック API

フロント（admin / public-viewer）は、これらの API を叩くだけで、
分析ロジックには一切関与しないという明確な役割分担になっています。

② バックエンド処理の時系列シーケンス図（PlantUML）

広聴AIの admin → API → LLM → DB → public-viewer の流れを、
実際の処理順に沿って図示します。

---

■ コメントアップロード → レポート生成 → 閲覧 までの全体シーケンス

@startuml

actor Admin
actor Viewer

participant "Admin UI\n(Next.js)" as AdminUI
participant "API Server\n(FastAPI)" as API
participant "LLM Client\n(OpenAI/Ollama)" as LLM
database "SQLite DB" as DB
participant "Public Viewer\n(Next.js)" as ViewerUI

== コメントアップロード ==
Admin -> AdminUI: コメントCSVを選択
AdminUI -> API: POST /reports/{id}/comments
API -> DB: コメントを保存
API --> AdminUI: {count: N}

== レポート生成開始 ==
Admin -> AdminUI: 「レポート生成」ボタン
AdminUI -> API: POST /reports/{id}/generate
API -> DB: タスク作成 (status=pending)
API --> AdminUI: task_id

== パイプライン処理 ==
loop 前処理
  API -> DB: コメント取得
  API -> API: テキストクリーニング
end

loop クラスタリング
  API -> LLM: コメント一覧を渡す
  LLM --> API: クラスタ結果
  API -> DB: クラスタ保存
end

loop 要約生成
  API -> LLM: クラスタごとのコメント
  LLM --> API: 要約
  API -> DB: 要約保存
end

API -> DB: レポート全体要約保存
API -> DB: タスク status=done

== 進捗確認 ==
AdminUI -> API: GET /tasks/{task_id}
API --> AdminUI: {status: done}

== 閲覧 ==
Viewer -> ViewerUI: レポート閲覧
ViewerUI -> API: GET /reports/{id}/clusters
API -> DB: クラスタ取得
API --> ViewerUI: クラスタJSON

ViewerUI -> API: GET /reports/{id}/summary
API --> ViewerUI: 要約JSON

@enduml

③ API と内部コード構造（router / service / repository）の対応

広聴AIのバックエンドは 3層構造になっています。

層	役割	例	
router	HTTP エンドポイント	/reports, /clusters	
service	ビジネスロジック	LLM 呼び出し、前処理、構造化	
repository	DB アクセス	コメント保存、クラスタ保存	


---

例：クラスタ取得 API のコード構造

/api/v1/reports/{id}/clusters


router

• リクエストを受ける
• service を呼ぶ
• レスポンスを返す


service

• DB からクラスタを取得
• 必要なら整形
• repository を呼び出す


repository

• SQLAlchemy で SQLite にアクセス
• clusters テーブルからデータ取得


---

まとめ

• API は レポート管理 / コメント管理 / クラスタ・要約 / パイプライン / タスク管理 の5系統
• レポート生成は 非同期タスクとして実行
• LLM（OpenAI/Ollama）は service 層で抽象化されている
• public-viewer は 完成済み JSON を取得して描画するだけ
• シーケンス図で見ると、admin → API → LLM → DB → viewer の流れが明確
