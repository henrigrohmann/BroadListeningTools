


1. パイプラインの目的と前提

目的

• 大量の自由記述テキスト（市民の声）を入力として受け取り、
“人間が読める構造化レポート” に変換する一連の処理フローを定義する。


前提

• 入力は CSV（または同等のテーブルデータ）
• 出力は Web UI（public-viewer 相当）が読める JSON / DB スキーマ
• LLM/embedding/クラスタリングは差し替え可能な Strategy として扱う
• 本番前に サンドボックス環境で検証・評価する


---

2. パイプライン詳細仕様

2-1. 全体フロー

[1] データ投入（CSV）
    ↓
[2] 正規化・前処理
    ↓
[3] embedding 生成
    ↓
[4] クラスタリング（階層 or フラット）
    ↓
[5] 濃いクラスタ抽出（密度ベース）
    ↓
[6] ラベル生成・要約（LLM）
    ↓
[7] レポート構造生成（JSON/DB）
    ↓
[8] 公開・閲覧（public-viewer）


---

2-2. 各ステージの仕様

[1] データ投入（ingestion）

• 入力: CSV ファイル（UTF-8）
• 必須カラム:• text: 市民の声（自由記述）

• 任意カラム:• topic_id, question_id, age, gender, region など

• 出力:• Idea オブジェクトの配列



{
  "id": "uuid",
  "text": "駅前のバスが少なくて不便です",
  "metadata": {
    "topicId": "T1",
    "questionId": "Q3",
    "age": 35,
    "region": "川崎市中原区"
  }
}


---

[2] 正規化・前処理

• 目的:• ノイズ除去・空行除去・極端に短い/長いテキストのフィルタリング

• 処理例:• トリム・改行統一
• 文字数閾値（例: 5文字未満は除外）

• 出力: フィルタ済み Idea[]


---

[3] embedding 生成

• 入力: Idea[]
• 処理:• text を embedding モデルに渡し、ベクトルを取得

• 出力:


{
  "id": "uuid",
  "text": "...",
  "vector": [0.0123, -0.98, ...],
  "metadata": {...}
}


• 要件:• バッチ処理対応
• モデル差し替え可能（OpenAI / SBERT / Llama embedding）



---

[4] クラスタリング

• 入力: Idea[]（vector 付き）
• 処理:• 距離関数: cosine
• アルゴリズム例:• AgglomerativeClustering（階層）
• HDBSCAN（密度ベース）


• 出力:


{
  "clusterId": "C1",
  "itemIds": ["id1", "id2", "id3", ...]
}


• 要件:• クラスタ数を自動決定（距離閾値 or density）
• 再現性（同じ入力→同じ結果）



---

[5] 濃いクラスタ抽出

• 入力: クラスタ一覧 + 各アイテムのベクトル
• 処理:• クラスタ内の平均距離 or 局所密度を計算
• 閾値以上を「濃いクラスタ」としてフラグ付け

• 出力:


{
  "clusterId": "C1",
  "isDense": true,
  "densityScore": 0.82
}


---

[6] ラベル生成・要約（LLM）

• 入力: クラスタごとの text リスト
• 処理:• LLM にプロンプトを投げてラベル・要約を生成

• 出力:


{
  "clusterId": "C1",
  "label": "交通の不便さ",
  "summary": "駅前のバス本数や乗り継ぎの悪さに関する不満が多いクラスタです。"
}


• 要件:• 日本語最適化
• 行政文脈で意味が通る表現
• ラベル長の制約（例: 5〜15文字）



---

[7] レポート構造生成

• 入力: クラスタ + ラベル + 濃さ + アイテム
• 出力: public-viewer 用 JSON / DB レコード


{
  "reportId": "R1",
  "title": "川崎市 市民の声レポート 2025-03",
  "clusters": [
    {
      "clusterId": "C1",
      "label": "交通の不便さ",
      "isDense": true,
      "densityScore": 0.82,
      "items": [
        {
          "id": "id1",
          "text": "駅前のバスが少なくて不便です",
          "metadata": {...}
        },
        ...
      ]
    },
    ...
  ]
}


---

[8] 公開・閲覧（public-viewer）

• 入力: レポート JSON / DB
• 機能:• クラスタ一覧
• 濃いクラスタの強調
• クラスタ詳細（要約＋サンプル発言）
• メタデータフィルタ（地域・年代など）



---

3. サンドボックスで同等のものを作るロードマップ

ここからが実務フェーズ。
「いきなり本番」ではなく、段階的に検証しながら作るロードマップを提案する。

---

フェーズ 0：スコープと前提の固定

目的:

• どこまでをサンドボックスで再現するかを決める


決めること:

• 対象データ（例: 過去のパブコメ 1,000件）
• モデル候補（embedding / LLM）
• 出力フォーマット（JSON スキーマ）
• 評価観点（例: クラスタの妥当性・ラベルの分かりやすさ）


---

フェーズ 1：最小パイプラインの実装（E2E）

ゴール:

• 小規模データ（例: 200件）で
「CSV → レポート JSON」まで一通り通る状態を作る


実装範囲:

• ingestion
• embedding（1種類でOK）
• クラスタリング（Agglomerative）
• ラベル生成（LLM）
• レポート JSON 出力（CLI or API）


検証:

• クラスタが「それっぽく」分かれているか（人間レビュー）
• ラベルが意味を成しているか
• 同じデータで再実行しても結果が安定しているか


---

フェーズ 2：クラスタリング・embedding の比較検証

ゴール:

• 「どの embedding / クラスタリング戦略が一番マシか」を把握する


やること:

• embedding を複数試す• OpenAI / SBERT / Llama embedding

• クラスタリングを複数試す• Agglomerative / HDBSCAN / KMeans

• 評価:• シルエットスコアなどの定量指標
• 人間レビュー（3〜5人でクラスタの妥当性を評価）



アウトプット:

• 「このユースケースなら embedding X + clustering Y が妥当」という暫定結論


---

フェーズ 3：濃いクラスタ抽出と UX 検証

ゴール:

• 「濃いクラスタ」の概念が本当に役に立つかを検証する


やること:

• 密度スコアの計算ロジックを実装
• 閾値を変えながら UI モックで確認
• 行政寄りのユーザー（想定）に見せてフィードバックをもらう


検証観点:

• 「濃いクラスタ」として強調されるものが、
本当に「重要そう」に見えるか
• 少数だが重要な声が埋もれていないか


---

フェーズ 4：LLM ラベル生成の安定化

ゴール:

• ラベルがブレすぎない状態を作る


やること:

• プロンプトのチューニング
• 温度・max_tokens の調整
• 同じクラスタに対して複数回生成して揺れを測定
• ラベル正規化ロジック（類似ラベルのマージ）検討


検証:

• 同じクラスタに対して 3 回ラベル生成 → どれくらい揺れるか
• 人間が「同じ意味」と判断できるか


---

フェーズ 5：public-viewer 相当の UI と結合

ゴール:

• 実際の閲覧体験を通じて、パイプラインの妥当性を検証する


やること:

• レポート JSON を読み込む簡易 viewer を実装（Next.js など）
• クラスタ一覧 → 詳細 → サンプル発言の流れを確認
• 濃いクラスタの強調表示
• メタデータフィルタ（地域・年代など）


検証:

• 「この UI なら、非エンジニアでも全体像を掴めるか」
• 「説明責任を果たせるだけの情報があるか」


---

フェーズ 6：負荷・スケール・安定性検証

ゴール:

• 実運用に耐えうるかを確認する


やること:

• データ件数を増やす（1,000 → 5,000 → 10,000）
• embedding / clustering の処理時間計測
• 並列化・バッチ処理の検討
• LLM API のレート制限・エラー時のリトライ設計


---

フェーズ 7：法務・ライセンス・運用ポリシーの整理

ゴール:

• 「このパイプラインを商用 or 実運用に載せて良い」と言える状態にする


やること:

• 使用モデルのライセンス確認（LLM / embedding）
• データ保持ポリシーの整理
• 匿名化の要否・方法の検討
• ログ・監査の設計


---

4. まとめ（あなた向けの一言）

• パイプラインの技術仕様は、もうかなりクリアに言語化できる段階に来ている
• 次の勝負どころは、• embedding / clustering / LLM の 戦略比較と検証プロセスの設計
• それを サンドボックスで安全に回すための段階的ロードマップ

• あなたのスタイルなら、• 各フェーズごとに「検証観点」と「合格ライン」を明示しておくと、
コントリビューターも巻き込みやすくなるはず
